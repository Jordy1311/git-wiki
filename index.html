Wh, wh and wh?

Git is officially defined as a distributed version control system (VCS).

In other words, it's a system that tracks changes to project files over time. It enables the user to record project changes and go back to a specific version of the tracked files at any given point in time. This system can be used by many people to efficiently work together and collaborate on team projects, where each person can have their own version of the project distributed on their computer. Later on, these individual versions of the project can be merged and adapted into the main version of the project. This aims to increase speed of development while maintaining all files' integrity.

Git was created in 2005 by Finnish-American software engineer Linus Torvalds, with contributions from other kernel developers, for the development of the Linux kernel. Since 2005, Junio Hamano has been the core maintainer of Git.

Design Characteristics

Git's design was inspired by BitKeeper and Monotone - alternative VSCs. It was originally designed as a low-level version-control system engine, on top of which others could write front ends, such as Cogito.

Strong support for non-linear development:

Git supports branching and merging and includes specific tools for visualising and navigating change history. A core assumption is that a change will be merged more often than it is written, as it is passed around to various reviewers.

Distributed development:

Git gives each developer a local copy of the full change history and changes are copied from one such repository to another.

Compatibility with existent systems:

Repositories can be published via HTTP (Hypertext Transfer Protocol), FTP (File Transfer Protocol) or a Git protocol over either a plain socket or ssh (Secure Shell Protocol) which means files are highly compatible.

Efficient handling of large projects:

Torvalds has described Git as being very fast and scalable, and performance tests done by Mozilla showed that it was an order of magnitude faster than some version-control systems; fetching version history from a locally stored repository can be one hundred times faster than fetching it from the remote server.

Cryptographic authentication of history:

The Git history is stored in such a way that the ID of a particular version (a "commit" in Git terms) depends upon the complete development history leading up to that commit. Once it is published, it is not possible to change the old versions without it being noticed.


How it works:

Projects are stored in repositories, which is just a container for project files with a few extra text files that are responsible for keeping track changes.

There are two types of repositories:

  • Local repository - this is one that is stored on one's own device and is generally linked to a remote. Any changes that are made to files stored on a local repository will not automatically reflect on the linked remote repository. This creates a development environment where users can make changes without having worry that their changes will negatively impact stable files which are stored on the remote. Likewise, changes that are made to the remote are not automatically reflected on one's own local repository, therefore, the remote needs to be "pulled" which updates the local to the same state as remote.
  • Remote repository - this is one that is stored on a server and is generally accessable to many people. In a project which involves many people, changes to these files should, in theory, only happen after someone has submitted the changes for review and they have been approved and merged. This is the aspect of Git that ensures integrity of files by making sure no one is responsible for changes.
Main/Master:

This is the set of files located on the remote or the files that are pulled off of the remote. If you are collborating with other developers and wanting to make changes or additions to the directory, you wouldn't usually make these changes or additions directly to the main/master as it means they will have no opportunity for review, instead you would make changes on another branch.

Pulling

Is the act of updating your local repository with the remote repository.

Branches and Checking-out:

A branch can be seen as an individual timeline of the project. With Git, you can create mutiple branches that allow you to add new (experimental, unfinished and/or potentially buggy) features in a seperate environment which can be submitted to others for review. Another member of a team could also create another branch off your branch in order to build on your ideas which adds a lot of thickness and complexity. Users can only be on one branch at a time. Switching branches will update your local repository from the files on the current branch to the files on the new branch. The act of changing branches is called "checking out **branch-x**".

Tracked/Untracked Changes:

Tracked changes are to files that were originally on the main/master branch. Untracked changes are files that are new to the project that Git has no record of.

Staging & Commits:

The staging area is an imaginary somewhere you can put changes before commiting them. This allows you to commit multiple changes that are related which helps clarify & classify a lot of work. Staging is the act of putting a change into this area. Committing is the act of recording a snapshot of the code as a whole (including changes that were staged).

Pull requests:

Note: this is different to pulling! Pull requests are a function of Github (a Git repository hosting service) that opens a ticket which details commits, files within those commits and differences (diffs) of those files if they were tracked, for the use of peer review. It allows one to request review of and communicate about changes that have been made.

Merges:

Once changes have been reviewed and approved by reviewers, merging is the next step. It takes the branch and combines it with main, making sure that any conflicts (places where code on both of the brances are different) are addressed. In simple cases, Github is able to deal with conflicts automatically, however, in more complex cases a "merge-conflict" may occur which will require user review to ensure code from both branches works as intended.

Definitions

  • Git - Is the name of the software describe above.
  • Repository - A remote directory that stores files managed by Git.
  • Master/Main - Is the main branch that stores all merged branches/initial files.
  • Branch - User created copy of master/main that allows the user to make changes without changing master/main.
  • Checkout - The act of changing where your changes will effect
  • Tracked/Untracked - Tracked changes are changes that are made to files already on master/main. Untracted are new files
  • Commit - Is made at user-defined stages to "save" changes to the branch/master/main.
  • Push request (PR) - When you have finished making changes on a branch and want to merge this back into master/main you will create a push request to submit to a reviewer.
  • Merge - The act of consolidating a branch with master/main.
  • Merge Conflict - Where changes conflict with another branch/master/main.

  • Stash - Temporarily storing changes while you make changes elsewhere.
